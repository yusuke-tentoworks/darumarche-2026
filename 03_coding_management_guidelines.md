# コーディング・管理ガイドライン（保守性）

このガイドラインは、時間が経っても、あるいは別の人が見ても「どこに何があるか分かる」「修正しやすい」状態を保つためのルールです。

---

## 1. ディレクトリ構造（フォルダ構成）

以下の構成を標準とし、プロジェクト間で統一します。

```text
/
├── index.html          # トップページ
├── about/              # 下層ページ（フォルダ分け）
│   └── index.html
├── assets/             # 静的ファイル
│   ├── css/            # スタイルシート（style.css, reset.cssなど）
│   ├── js/             # JavaScript（main.js, vendor/など）
│   ├── images/         # 画像ファイル（ページごとにフォルダ分けも検討）
│   ├── fonts/          # Webフォント
│   └── pdf/            # 資料などのPDF
└── .env                # 環境変数（Gitには含めない）
```

---

## 2. 命名規則（名前の付け方）

### 2-1. ファイル・フォルダ名
- **基本**: `小文字英数字` と `ハイフン(-)` のみ。
- **NG**: 日本語、スペース、大文字、アンダースコア（_）は原則避ける（URLに影響するため）。

### 2-2. CSSクラス名
- **役割がわかる名前にする**: `.header`, `.footer`, `.btn-primary` など。
- **BEM（風）の採用**: コンポーネント構造を意識する。
    - `[ブロック]__[要素]--[状態]`
    - 例: `.card`, `.card__title`, `.card__btn--active`

---

## 3. HTML/CSSの書き方ルール

### 3-1. HTML
- **インデント**: スペース2つまたは4つで統一する。
- **セマンティック意識**: 意味のない `<div>` の多用を避け、`<section>`, `<article>`, `<aside>`, `<nav>` を適切に使う。

### 3-2. CSS
- **コメントでの区分け**: 
  ```css
  /* ==========================================
     Header
     ========================================== */
  ```
- **変数の活用**: 色やフォントサイズは CSS変数（`--main-color`など）で管理し、一括変更しやすくする。

---

## 4. コンポーネント化と再利用

- **共通パーツの切り出し**: ヘッダーやフッターなど全ページ共通の部分は、AIに対して「共通パーツとして読み込める方式（PHPの include や JS での静的挿入など）で作成して」と指示する。
- **スタイルの一貫性**: 似たようなボタンを複数作らず、1つの汎用クラス（`.btn`）を作って使い回す。

---

## 5. バージョン管理（Git）

- **コミットメッセージ**: 何を変更したか簡潔に記述する。
  - `feat: ○○機能の追加`
  - `fix: △△の表示崩れを修正`
  - `docs: ガイドラインの更新`

---

## 6. プロジェクトの整理（クリーンアップ）

プロジェクトを常に清潔に保ち、ミスや脆弱性を防ぎます。

### 6-1. 未使用ファイルの削除
- **不要な画像・素材**: デザイン変更で使わなくなった画像やロゴ、テスト用の素材は速やかに削除する。
- **未使用のコード**: どこからもリンクされていない `test.html` や、コメントアウトして残してある巨大な古いコードは残さない。

### 6-2. 本番環境に残さないファイル
- **バックアップファイル**: `index_20240412.html` のような日付入りの古いファイルは、Gitで管理しているため本来不要です。混同を避けるため削除します。
- **サンプル・ドキュメント**: 外部ライブラリを導入した際に付いてくる `README` や `docs` フォルダ、サンプルコードが公開サーバーに残らないようにする。

---

## 7. AIへの指示出しルール（保守性のために）

制作中に Antigravity に以下を意識させてください。

> **「将来的にメンテナンスしやすいように、色やサイズをCSS変数にまとめて、共通パーツは再利用可能な形でコードを書いてください。」**
> **「ディレクトリ構造はこのプロジェクトの標準ルール（coding_management_guidelines.md）に従ってください。」**
